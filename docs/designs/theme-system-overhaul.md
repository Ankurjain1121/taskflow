## Problem Statement

TaskFlow has a single hardcoded "Warm Earth" theme. The entire color system — CSS custom properties, PrimeNG bridge, task priority/status colors — is baked to one palette. Users cannot personalize their workspace appearance. Meanwhile, the earth-tone data visualization colors (priority badges, kanban cards, column headers) lack the instant readability of conventional task management colors (red = urgent, green = done, blue = in progress, yellow = medium).

Two independent problems:
1. **Data viz colors** are too muted — earth tones don't communicate priority/status at a glance
2. **Website theming** is locked to one palette — no personalization, no differentiation

## Constraints

- Must work with existing PrimeNG component library (requires `definePreset()` JS calls)
- Must maintain WCAG AA contrast ratios across all theme combinations
- Must persist user's theme choice to backend (`user_preferences` table — repurposing `accent_color` for light theme, adding `dark_theme` column for dark theme)
- Must support cross-tab sync (existing `localStorage` + `storage` event pattern)
- Dark mode must remain independently toggleable (light/dark/system) regardless of theme
- Existing CSS custom property **names** (`--background`, `--foreground`, `--card`, etc.) must not change — only their values per theme
- **Attribute migration:** `data-accent` attribute is retired and replaced by `data-theme` (light theme) and `data-dark-theme` (dark theme). All references to `data-accent` must be updated, including the inline FOUC-prevention script in `index.html`
- Backend `VALID_ACCENT_COLORS` (currently 9 entries: indigo, blue, green, orange, rose, violet, amber, slate, earth) will be replaced with 7 new light theme IDs. A separate `VALID_DARK_THEMES` list (2 entries) will be added

## Premises

1. Data viz colors are universal — same red/green/yellow/blue regardless of active website theme
2. Repurpose existing `accent_color` column as light theme identifier; add new `dark_theme` column via migration
3. Users pick light theme + dark theme independently (7 light options, 2 dark options)
4. Blast radius is well-contained: ~8 key files + backend migration

## Approaches Considered

### Approach A: Pure CSS (data-theme attribute selectors)
All 9 palettes defined as CSS custom property blocks. ThemeService sets `data-theme` attribute. CSS does the rest.
- **Pro:** Pure CSS switching, instant apply, easy to add themes
- **Con:** PrimeNG still needs JS `definePreset()`, styles.css grows significantly
- **Effort:** M

### Approach B: Full TypeScript Registry
All palettes as TypeScript objects in `theme-palettes.ts`. ThemeService applies CSS vars via JS + calls `definePreset()`.
- **Pro:** Single source of truth, type-safe, CSS stays lean
- **Con:** Brief FOUC on switch, harder to inspect in DevTools
- **Effort:** M

### Approach C: Hybrid (CSS + TS bridge)
CSS attribute selectors for semantic variables, TypeScript objects only for PrimeNG ramp. ThemeService sets `data-theme` + calls `updatePrimeNG()`.
- **Pro:** Instant CSS switching for 95% of UI, JS only for PrimeNG, DevTools-inspectable
- **Con:** Two places to update per new theme
- **Effort:** M

## Recommended Approach

**Approach C: Hybrid** — chosen for debuggability, instant switching, and clean separation of concerns.

### CSS Cascade / Specificity Order

This is critical. The cascade must be:

```css
/* 1. Base light (lowest specificity — :root = 0,0,1) */
:root { --background: #EDE9DD; }

/* 2. Light theme overrides (0,1,0 — attribute selector) */
[data-theme="sea-foam"] { --background: #c9dce1; }

/* 3. Dark mode base (0,1,0 — class selector, BUT declared AFTER
      [data-theme] blocks so it wins on equal specificity) */
html.dark { --background: #1C1A17; }

/* 4. Dark theme variant (0,2,0 — class + attribute, highest) */
html.dark[data-dark-theme="purple-night"] { --background: #171821; }
```

**Key:** `html.dark` has specificity (0,1,1) — one class `.dark` plus one element `html`. `[data-theme="X"]` alone has specificity (0,1,0). So `html.dark` wins by specificity, not just source order. The light theme's `data-theme` attribute stays on `<html>` but its CSS vars are masked by the `html.dark` block. When dark mode is turned off, the `dark` class is removed and the light theme's `[data-theme]` vars become visible again.

### Dark Theme Strategy

Two curated dark palettes (Warm Earth Dark + Purple Night). ALL light themes share these same two dark options. This avoids a 7x2 matrix of unique dark palettes.

When dark mode activates:
1. `html.dark` applies default dark vars (Warm Earth Dark) — declared AFTER all `[data-theme]` blocks, so same-specificity source order wins
2. If `data-dark-theme="purple-night"`, `html.dark[data-dark-theme="purple-night"]` has higher specificity (0,2,0) and overrides
3. The light theme `data-theme` attribute stays on `<html>` but its vars are masked by the dark cascade. When dark class is removed, the light theme vars become visible again

**PrimeNG debounce note:** `updatePrimeNG()` should track the last-applied theme identity (a string like `"sea-foam:light"` or `"purple-night:dark"`) and skip the `definePreset()` call if the identity hasn't changed. This prevents redundant recalculations during rapid dark/light transitions.

## Implementation Details for Expansion Items

### 1. Animated Theme Transitions
- In `theme.service.ts`: `setLightTheme()` and `setDarkTheme()` add `.theme-transitioning` to `<html>`, then remove it after 200ms via `setTimeout`
- The `.theme-transitioning` CSS class (already in `themes.css:87-95`) applies `transition: background-color 200ms, color 200ms, border-color 200ms`
- Respects `prefers-reduced-motion: reduce` (already handled in themes.css)

### 2. Command Palette Theme Switcher
- File: `frontend/src/app/shared/components/command-palette/command-palette.component.ts`
- Add a "Switch Theme" command to the existing command registry
- On selection, show sub-list of 7 light + 2 dark swatches (colored squares + name)
- Each swatch calls `ThemeService.setLightTheme()` or `setDarkTheme()`
- Command appears when user types "theme", "color", or "appearance"

### 3. Theme Preview on Hover
- File: `frontend/src/app/features/settings/appearance-section/appearance-section.component.ts`
- `(mouseenter)` on swatch card: call `ThemeService.previewTheme(theme)` — sets `data-theme` attribute WITHOUT saving to localStorage or backend
- `(mouseleave)` on swatch container: call `ThemeService.revertPreview()` — restores committed theme from signal
- `(click)` on swatch card: call `ThemeService.setLightTheme(theme)` — commits the preview
- `ngOnDestroy()`: call `ThemeService.revertPreview()` as safety cleanup
- **Debounce:** 150ms delay before applying preview to avoid jank from rapid mouse movement across swatches
- PrimeNG `definePreset()` is NOT called during preview (CSS-only) — only on commit. This avoids expensive recalculations during hover.

### 4. Per-Theme Sidebar Colors
- Each `ThemePalette` in `color-palettes.ts` includes a `sidebar` sub-object:
  ```typescript
  sidebar: { bg: string; surface: string; surfaceHover: string; surfaceActive: string;
             border: string; textPrimary: string; textSecondary: string; textMuted: string; }
  ```
- Colors are derived from each theme's own 6-color ramp (user requirement: "sidebar must use each theme's color")
- ThemeService applies `--sidebar-*` CSS vars via the `[data-theme]` CSS blocks (not via JS)
- Dark themes have their own sidebar colors in the `html.dark[data-dark-theme]` CSS blocks

### 5. Per-Theme Font Pairing
- Each `ThemePalette` includes `fontDisplay` and `fontBody` fields
- Proposed pairings (to be finalized during design review):
  - White Heaven: Inter (display) + Inter (body)
  - Sea Foam: Nunito (display) + Nunito Sans (body)
  - Warm Earth: Syne (display) + DM Sans (body) — current
  - Storm Cloud: Space Grotesk (display) + Work Sans (body)
  - Morning Sky: Poppins (display) + Poppins (body)
  - Misty Forest: Lora (display) + Source Sans 3 (body)
  - Modern Dental: Outfit (display) + Outfit (body)
- All fonts preloaded in `index.html` (zero FOUT on switch, ~150KB one-time cost)
- ThemeService sets `--font-display` and `--font-body` CSS vars on theme switch
- `font-display: swap` on all @font-face rules for fallback during initial load
- Self-hosting fonts (instead of Google Fonts CDN) recommended for corporate firewall resilience — to be evaluated during eng review

1. **`:root` specificity is (0,1,0) not (0,0,1)** — cascade reasoning still holds but design doc's math should be corrected
